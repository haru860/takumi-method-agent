# 重要な注意事項

## 入力ファイルの優先

**検証する内容は、必ず入力ファイル（`output/proposal-format*.md`、`input/idea.md`）の実際の内容に基づくこと。**

- 以前のセッションの情報や、他の企画の内容を使用しない
- サンプルデータや例示の固有名詞は使用しない
- 入力ファイルに記載されていない内容を検証してはならない

## 手順2: 企画書の分析

### 企画書とは

匠Methodの成果物（Phase 1〜4）を基に生成されるビジネス企画書。Why-What-How型（format1）とストーリーテリング型（format2）の2フォーマットがあり、入力ファイルの内容に忠実に構成される。

### 2-1: 良い点の分析（5つ）
以下の観点から企画書の強みを評価する：
- **入力ファイルとの整合性**: 企画名・ステークホルダー・ビジョン・コンセプトが入力ファイルと一致しているか
- **構成の論理性**: セクション間のつながりが論理的で説得力があるか
- **エグゼクティブサマリーの完成度**: 企画全体を簡潔に伝えられているか
- **ステークホルダーへの価値の具体性**: 各ステークホルダーへの価値提案が具体的に記載されているか
- **実現戦略の具体性**: 業務要求・IT要求・活動が具体的に記載されているか
- **フォーマットの適切さ**: 選択したフォーマット（Why-What-How型/ストーリーテリング型）の特性が活かされているか
- **Phase 1-4の網羅性**: すべてのフェーズの成果物が適切に反映されているか
- **読みやすさ**: レイアウト・表現が読み手にとってわかりやすいか

### 2-2: 悪い点（改善点）の分析（5つ）
以下の観点から改善すべき点を指摘する：
- **入力ファイルとの不整合**: 企画名・ステークホルダー・ビジョン等が入力ファイルと一致していない
- **創作・補完の混入**: 入力ファイルに記載のない情報が追加・創作されている
- **セクション間の断絶**: セクション間のつながりが不自然で論理的な流れが途切れている
- **エグゼクティブサマリーの不足**: 企画全体を把握するには情報が不十分
- **価値提案の抽象性**: ステークホルダーへの価値が抽象的で具体性に欠ける
- **実現戦略の不足**: 業務要求・IT要求・活動の記載が不十分
- **フォーマット特性の未活用**: フォーマットの特性（論理的構成/物語性）が十分に活かされていない
- **レイアウトの問題**: ページからのはみ出し、情報過多、視覚的なバランスの悪さ

## 手順3: 企画書を改善するための問いの生成

**重要**: 問いは、手順2で分析した「良い点」と「改善点」に直接紐づけて生成すること。

### 問いの生成方針

5つの問いを以下のように構成する：

1. **改善点1に対応する問い**: 改善点の1つ目を解決するための問い
2. **改善点2に対応する問い**: 改善点の2つ目を解決するための問い
3. **改善点3に対応する問い**: 改善点の3つ目を解決するための問い
4. **良い点を活かす問い**: 良い点をさらに強化・発展させるための問い
5. **良い点と改善点を統合する問い**: 良い点を維持しながら改善点を解消するための問い

### 問いの形式

各問いは以下の形式で出力する：

```
1. 【改善点「○○」に対応】問いの内容...
2. 【改善点「○○」に対応】問いの内容...
3. 【改善点「○○」に対応】問いの内容...
4. 【良い点「○○」を活かす】問いの内容...
5. 【良い点「○○」×改善点「○○」の統合】問いの内容...
```

### 生成のポイント

- 答えやすい問いではなく、**深く考えさせる問い**にする
- 「はい/いいえ」で答えられない**オープンクエスチョン**にする
- **分析した良い点・改善点と明確に対応**させる
- 回答することで**企画書が具体化・深化する**問いにする
